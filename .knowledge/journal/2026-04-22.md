### 2. 修复细节

#### 2.1 Bug 1 — `LLM.get_offload_diagnostics()` 对部分初始化实例要容错

`c816a9c`（P2）给 `LLM` 加了 `self.expert_map_store` 字段，但 `get_offload_diagnostics`
里用 `self.expert_map_store.diagnostics()` 直接访问。现有单测为了不走完整
`AutoConfig.from_pretrained → safetensors` 的加载路径，用 `LLM.__new__(LLM)` 绕过
`__init__`，只手工 set 了 diagnostics 需要的少数字段，**缺了新增的 `expert_map_store`
导致 `AttributeError`**。

修复：用 `getattr(self, "expert_map_store", None)` 替换。与同文件 `reset_offload_diagnostics`
里其它字段的访问风格一致，并保持对"部分初始化"的向后兼容。

#### 2.2 Bug 2 — `test_backend_notify_expert_evicted_called_on_demotion` 重写

`04dfbda` 加入的这条测试用了一堆**根本不存在的 API**：

- `from nano_ktrans.utils.context import InferenceContext` — `context.py` 里类名叫 `Context`
- `HybridMoE(expert_hidden_size=..., expert_intermediate_size=..., expert_key_template=..., expert_proj_names=..., offload_backend_name=...)` — 真实签名是 `hidden_size / moe_intermediate_size / offload_backend`；`expert_key_template / expert_proj_names` 是可选 kwargs 但名字对不上
- `moe.update_residency_plan(...)` — `HybridMoE` 上根本没有这个方法

显然是 AI 生成但**从未实际跑过**的测试。修复思路是参照相邻真实测试
`test_hybrid_moe_applies_decode_migration_plan` 的模式重写：

1. 直接构造 `HybridMoE(num_experts=, top_k=, hidden_size=, moe_intermediate_size=, gpu_experts=ModuleDict(), gpu_experts_mask=, ...)`
2. 通过 `hybrid.offload_backend.queue_migration_plan([ExpertMigrationOp(src=GPU, dst=PIM, ...)])` 下发一个 demotion op
3. forward 时迁移管线会调用 `_demote_expert_from_gpu` → `notify_expert_evicted(expert_idx, 'gpu')`
4. 用 tracked wrapper 验证确实被调用、`residency_before == "gpu"`、`gpu_experts_mask[1] == False`

保留了原测试的意图（验证 `04dfbda` 修的 DPU 权重回收通知真的被调用），同时把
API 换成真实存在的。

#### 2.3 Bug 3 — `ExpertWeightLoader` 允许空 `weight_path`

`HybridMoE.__init__` 在构造末尾**无条件**实例化：

```python
self.materialization_manager = ExpertMaterializationManager(weight_path=weight_path, ...)
```

`ExpertMaterializationManager.__init__` 又无条件 `self.loader = ExpertWeightLoader(weight_path)`，
而 `ExpertWeightLoader.__init__` 会立即扫描 `*.safetensors`，没有就抛 `FileNotFoundError`。

`test_cpu_only_smoke_generation_path` 是合法的单元测试用例：纯 GPU 专家 + 随机初始化
权重 + `weight_path=""`，**根本不需要加载任何 safetensor**，却因为这条硬性检查无法
构造模型。这个回归自 `ExpertMaterializationManager` 引入后就存在（smoke test 自
`047af5c` 加入后从未更新过），本地从来没跑过 smoke test 才一直没爆出来。

修复（最小侵入）：让 `ExpertWeightLoader.__init__` 在 `weight_path == ""` 时进入
「空加载器」状态 —— `_files=[]`、`_key_to_file={}`、`_quantize_config={}`，不抛错。
实际 `load_*` 方法被调用时，原有的 `KeyError("Weight key '...' not found")` 会自动
抛出，诊断信息反而更具体。**非空路径但缺 safetensor 的情况仍然抛 `FileNotFoundError`**，
不掩盖 typo。

## 踩坑与经验

（四条新 gotchas 已写入 `.knowledge/context/gotchas.md`）

1. **`LLM.__new__(LLM)` 绕过 `__init__` 的测试模式要求 diagnostics 方法必须字段容错**。
   新增任何在 `__init__` 里才赋值的公开字段，凡是在 `get_*` / `reset_*` 诊断路径上
   被访问的，都应该用 `getattr(self, "field", None)` 而不是 `self.field`。

2. **AI 生成的测试必须本地实跑一次才能合入**。`04dfbda` 的 broken test 和
   `c816a9c` 之后的 `test_llm_get_offload_diagnostics_*` 都是"只过了 review 没过
   runtime"的典型。建议在 `pyproject.toml` 里 pin 一个 CI hook，或者把 pytest
   结果写进 PR 描述。

3. **"硬依赖 weight_path 非空"的构造器会破坏纯随机初始化的 smoke test**。凡是
   `HybridMoE / MixtralModel` 这种既要支持"加载真实权重"又要支持"随机初始化跑通
   forward"的类，其子组件（loader / materialization manager）在构造时**不应该**
   对 weight 文件存在性做强校验，改到真正需要加载时再抛错。

4. **只有 smoke test 才能暴露"整条类链路能否默认参数构造"的回归**。单元测试逐
   组件构造时都手工给了合法 `weight_path`，掩盖了 HybridMoE 本身对空 `weight_path`
   的假设。应把 smoke test 纳入 CI 必跑集。

## 5. 下一阶段科研目标设定（11:10）

用户要求明确"PIM 全线超过 CPU"的科研工作方向。系统梳理了：

- `ADR-001` 的 6 个可借鉴创新点（P1/P2 已落地，P3-P6 待做）
- `2026-04-20` 的 5 个关键差距（其中 Gap 2 "T-MAC bit-serial kernel" 在 commit `169ea55` 被声称为"已实现"）
- 代码实际状态：`dpu_quantized_kernel.c` 里 `kernel_mode=6` 被标为 T-MAC，但**实际只是 activation 侧朴素 shift-add 乘法模拟**，没有真正的 LUT 编码 weight，DPU 软件乘法依然在内循环上

基于这些发现，落档 `ADR-002`（PIM 算子级超越 CPU 的优化路线），4 个里程碑：

| M | 名称 | 周期 | 关键交付 |
|---|------|------|---------|
| M-1 | 基线建立 | 1-2 天 | 修 e2e `gate_up_proj` bug + 全 shape×batch×rank×mode sweep |
| M-2 | 真 T-MAC kernel | 1 周 | `kernel_mode=7`：weight bit-sliced LUT + activation bit-pack 索引，DPU 内循环零乘法 |
| M-3 | Cost model + overlap | 1 周 | `BackendCostModel`（ADR-001 P3+P6）+ async PIM submit，形成真正的 GPU/PIM overlap |
| M-4 | 研究探索 | 2-3 周 | Mixed precision、LUT 共享、sub-batch interleaving |

**核心技术洞察**：

1. **DPU 没有硬件乘法器** — 每条 `int8 × int16` 是 ~10 cycles 的软件实现，是当前量化 PIM 路径的**根本瓶颈**。所有不消除乘法的优化（tile、batch、LUT 精度调整）都只能改常数项。

2. **T-MAC 的本质是"把 weight 完全编码进 LUT，内循环纯查表"**，不是"把乘法拆成 shift-add"。当前 `kernel_mode=6` 搞反了——它拆的是 activation 而不是 weight，所以 weight 部分的 `lut[q]` 读取还在，而内循环又多了 7 轮 shift-add，反而**可能更慢**。

3. **算子层面赢了才有端到端的机会**。当前端到端 `cuda_pim` 因为 `gate_up_proj` checkpoint layout 适配 bug 根本没跑过，所有"PIM 赢"的证据仅限 operator-only micro benchmark。**M-1 必须先建立可信的端到端基线**。

4. **Cost model 是兜底**。即使 T-MAC 在某些 shape 下赢不了 CPU（特别是 `batch=8` 的 `gate` 形状），系统也要能自动路由到 CPU，不能硬撑到 PIM 然后输。

### 6.2 M1-T2 — 新增 sweep 脚本

`benchmarks/benchmark_pim_shape_sweep.py`，按 ADR-002 §5.1 的 Level-2 基准设计：

- Qwen3 真专家形状 `{gate, up, down}` × batches `{1, 2, 4, 8}` × ranks `{1, 4, 8, 16, 32}` ×
  kernel_modes `{3, 4, 6}` = 默认 180 cell
- 每 cell 同时记录 CPU grouped / CPU dense 两条基线 + PIM 的 `launch/transfer` breakdown
- 一级 KPI：`pim_vs_cpu_grouped_ratio`（= cpu_grouped_avg / pim_avg，越大越好）
- 二级 KPI：`max_abs_error_vs_cpu_grouped`（ADR-002 要求 ≤ 0.05 相对满量程）
- 每 (shape, batch) 自动汇总 `best_kernel_mode` / `best_rank_count`
- 失败隔离：PIM cell 初始化失败或 kernel 拒绝形状时记 `{"status":"pim_error", "pim_error":...}`，
  不阻塞整表
- 默认走真实 `Qwen3-30B-A3B-GPTQ-Int4`，可 `--synthetic` 用随机权重做 smoke
- 输出文件强制走 `--json-out`，并实时 echo 每 cell 的单行进度（长 sweep 可观察）

本地会话没有 `/dev/dpu_rank*`，只跑了 `--help` 验证 CLI；真机 sweep 留给宿主机。

### 6.3 M1-T3 — mode=6 静态审计（防止"T-MAC 标签欺骗"固化）

读 `dpu_quantized_kernel.c` 的 `kernel_mode == 6` block，直接看到：

```c
const int32_t w0 = (int32_t)lut0_i16[q0];   // 权重还在被 nibble-key 查表
const int32_t w1 = (int32_t)lut1_i16[q1];
const uint32_t abs_x = (uint32_t)(x >= 0 ? x : -x);
if (abs_x & 0x01u) { partial0 += w0;       ... }   // activation 侧 shift-add
if (abs_x & 0x02u) { partial0 += w0 << 1;  ... }
... 一路到 0x40u
```

这是 **activation 侧逐位 shift-add 乘法器**，不是 T-MAC。真正的 T-MAC 要求
**weight 完全编码进 LUT 索引，内循环纯查表**（`T[j][pack(x[j*g..])]`）。

把这一事实写成三条静态断言（`TestQuantizedKernelAudit`）：
1. mode=6 内循环必然出现 `lut0_i16[q0]` 和 `lut1_i16[q1]`（= 权重还没编码进索引）
2. mode=6 内循环必然出现 7 个 `abs_x & 0xNN` 模式（= activation bit-scan）
3. mode=4 保持单个 `x * (int32_t)lut0_i16[q0]` 作 honest baseline

**这三条断言就是"M-2 落 mode=7 时必须替换的审计标记"**—— M-2 改完后删这三条，
换成 mode=7 的数值正确性测试。

## 7. dev_gate：数据驱动的 milestone 守门人（11:50）

用户要求："程序需要首先读取并验证上一次处理的数据，检查其是否完全满足预设的规范和质量要求。
如果上一轮数据通过验证且新数据已就绪，则进入开发阶段；否则，暂停流程并报告具体的检查结果或等待新数据到达。"

落地 `scripts/dev_gate.py`（~670 行，零第三方依赖），作为 ADR-002 milestone 推进的
**唯一决策点**。三阶段串行：

1. `prerequisite_check` — 前置 milestone 必须是 PASS 才继续
2. `artifact_check` — artifact 必须存在且 mtime 严格新于上次 PASS 的 snapshot
   （防止"上次的数据又被拿来当这次的证据"）
3. `acceptance_check` — 按 `path/op/value` 规则逐条评估

实现上几个有意思的决策：

- **幂等 + 防重放**：PASS 时记下 artifact mtime；下次评估如果文件没变就 WAIT，
  不允许静默复用旧判决
- **`acceptance_checks: []` → BLOCKED**：防止 spec 留空就自动 PASS 这种漏洞
- **聚合路径语法**：支持 `min(results[*].x)` / `max(...)` / `count(...)`，
  刚好贴合 sweep artifact 的 shape（"所有 cell 的 min ≥ 0.9x"）
- **零耦合**：不 `import nano_ktrans`，只读 JSON，主代码重构期间也能独立工作
- **可审计 bypass**：`dev_gate bless --force --note "..."` 是唯一手动标 PASS 的路径，
  且 state.json 和 log 都会标 `bypassed: true`

注册了 M-1 和 M-2 两个 spec：
- M-1 要 e2e smoke `status==ok` + sweep 真数据（`use_synthetic==false`）+ 至少一个
  cell PIM ≥ CPU（证明测量基础设施可信）
- M-2 的 `prerequisites=["M-1"]`，`pim_vs_cpu_grouped_ratio ≥ 1.5x`（ADR-002 KPI），
  `min(...) ≥ 0.9x`（无 batch cliff），`max_abs_error ≤ 0.05`

本地真跑一次 `check`：M-1 → **WAIT**（artifact 没有，列出宿主机命令），
M-2 → **HALT**（M-1 没 PASS 不评估），exit=1。正好是用户要求的行为。

测试：新增 23 条 `test_dev_gate.py`，覆盖路径解析 / 规则评估 / 五种 verdict /
freshness / prereq 链。全量 `pytest tests -q` = **183 passed, 1 warning**。

### 8.2 M1-T1 下游级联 bug 连击

即使 ADR-002 M-1 提交里已经修了 `adapt_config_to_checkpoint` 的 `.qweight`
探测，跑 e2e 还是接连撞上三个级联 bug：

**级联 1** — `CPUMoEBackend.__init__` 忽视 `is_gptq` 标志：
`_detect_and_load_gptq` 成功后仍然调 `load_layer_experts_stacked(...)` 去
找 `.weight`。修复是引入 GPTQ 早退，并给几个关键 entry point（
`export_expert_weights / _compute_expert_output_cpu / submit_forward /
sync_forward`）补 "GPTQ without cpu_infer" 的守卫路径，让它们在 GPTQ 模式
下不要去访问 fp16 stacked tensors 或 C++ AMX engine，而是让 subclass
（PIMMoEBackend）来接管。

**级联 2** — `ExpertWeightLoader` 每层重建 index：
Qwen3 48 层 × 每层 2 处构造 loader，每次都要打开全部 safetensor 文件、
iterate 74739 个 key。首次冷启动观测到 25 min 仍没加载完。上了进程级
`_index_cache`，键是 `(abs_path, (file, mtime_ns))`，同一 checkpoint 内
只扫一次。

**级联 3** — `safe_open(...).get_tensor()` 每次都 mmap：
真正的杀手。`load_gptq_expert_linear` 对每个专家做 4 次 `_load_tensor`，
每次 `with safe_open(file)` 都是一次新的 mmap。观测到**Layer 0 单层
192 秒**。加了进程级 `_open_handle_cache`，单个 safetensor 文件只
mmap 一次、在进程生命期内保持打开。加速 150×（每层 1.3s）。

3 个 bug 全修完才真正跑通 e2e smoke；第三次运行的输出：

## 9. M-2 执行与负结果（同日晚些时候）

### 9.1 实施

按 ADR-002 §4.2 规划实现 `kernel_mode=7` 真 T-MAC bit-serial 核：

- **Host 端 bit-plane 预处理**：把 int8 activation 每 `BLOCK_FLOATS=64` 的 block 拆成 8 个 `uint64_t` bitmasks（7 magnitude planes + 1 sign plane），经 `dpu_broadcast_to("inputs_bitplanes_mram", ...)` 传 DPU。
- **DPU 内循环**：weight LUT 每元素查一次并把 activation sign 合入 `w_block_heap[]`；然后按 bit-plane 外展，仅扫 set bits（`__builtin_ctz` 32-bit 半字扫描）累加到 `S[bp][0..1]`；最终 `acc = Σ_b S[bp] << b`。**零软件乘法**、**零 per-element shift loop**。
- **WRAM 栈修复**：第一次跑 `api:W: One or more DPUs are in fault`，root cause 是新加的 `int32_t w0/w1_block[64]` + `bp_cache[8]` 栈 VLA 把 `STACK_SIZE_DEFAULT=2048` 顶爆。改用 `mem_alloc`（WRAM heap）解决。

### 9.2 数据（`pim_shape_sweep_M2_tmac.json`, 120 cells）

| mode | ratio_max | ratio_min | mean | max_err |
|------|-----------|-----------|------|---------|
| 4 | **3.32×** | 0.42× | **1.45×** | 0.424 |
| 7 | 1.15× | 0.12× | 0.48× | 0.424（与 mode=4 **bit-exact**）|

**mode=7 在 0/60 个 cell 跑赢 mode=4**。

### 9.3 关键认识

DPU 的 `int8×int16` 软件乘法 **已被 UPMEM SDK 优化到 ~10 cycles**，而 DPU **没有 hw ctz/popcnt**。
用 bit-plane 条件加法（+ 软件 ctz）替换乘法，**节约不了 cycles**，反而要付 per-block bit-plane DMA 和 weight unpack 的额外开销。

这与 T-MAC 论文（ARM/x86）的结论形成对照：T-MAC 收益本质来自 "SIMD-friendly LUT 替代混合精度 MAC"，UPMEM DPU 既无 SIMD 也无 ctz，所以收益为负。

**这是 publishable negative result**，已写入 `ADR-002 §10`。

### 10.2 顺带修复的 M-1 遗留 bug

真机 e2e 第一轮发现：`cuda_cpu_offload` 跑 GPTQ checkpoint 时 decode
10 秒跑完 32 tokens，看似非常快。抓 diagnostics 发现 `is_gptq=None,
use_fallback=False, fallback_output=zeros` —— CPU 路径根本没在计算。

根因：`CPUMoEBackend.submit_forward` 在 `is_gptq and cpu_infer is None`
时直接 `self._fallback_output = torch.zeros_like(hidden_states); return`。
M-1 当时为了让 GPTQ + 无 AMX 的测试不崩，这是对的；但生产 decode 就
把整个 MoE 层变成 no-op，任何基于这条路径的 TPS 数字都是假的。

修复：`CPUMoEBackend._compute_expert_output_cpu_gptq(states, cpu_slot)`
调 `cpu_w4a32_matvec(states, expert_gptq[proj])` 真做 CPU grouped forward；
`submit_forward` 在"子类没先填 `_fallback_output`"时走这条新路径。

修完后 cuda_cpu_offload 32 token decode 从 4s 跳到 10.4s，**符合真实
W4A32 grouped 成本**。
