### 仍然阻塞最终加速目标的 5 个关键差距

**Gap 1: 量化权重常驻 — quantized preload/infer 尚未实现**

当前 `host_expert_bridge.c` 的 preload/infer 仅支持 fp32 expert 路径。
`host_quantized_bridge.c` 的 `pim_quantized_linear_run()` 仍是"每次调用都传输 qweight + scales + LUT"的旧模式。
量化路径才是真实部署路径（GPTQ-Int4），fp32 路径只是功能验证。

**→ 需要：为 quantized bridge 也实现 preload/infer 拆分，让 qweight + scales + LUT 常驻 MRAM。**

**Gap 2: batch>1 量化性能悬崖 — 需要 T-MAC 式算法重构**

当前 `kernel_mode=4` 在 batch=1 上已超过 CPU（1.30-2.54x），但 batch>4 反而更慢（0.56x）。
根本原因：当前内循环是 `x * lut[q]` 逐元素乘累加，batch 增大时 input 读取量线性增长。

T-MAC 论文的核心思路可以解决这个问题：
- **位串行分解 + 预计算部分和表**：把 4-bit 权重拆成 1-bit 切片，每个切片的所有 2^1=2 种可能的部分和预计算存表
- **表查找替代乘法**：内循环变成 `acc += LUT[bit_slice]`，完全消除乘法
- **对 DPU 极度友好**：DPU 没有硬件乘法器，所有乘法都是软件模拟；表查找是纯内存操作，正是 PIM 的强项

**→ 需要：新增 kernel_mode=6（T-MAC bit-serial LUT），重新设计 DPU 内循环。**

**Gap 3: 逐 expert 的 Python 控制开销**

`_submit_forward_real()` 中 `for expert in activated_cpu_experts` 是逐个处理。
`_apply_promotion_batch()` 中 resident set 注入也是逐 expert。
decode 关键路径上 Python 循环开销随专家数增长。

**→ 需要：batch 化 GPU module 注入（`.update()` + 统一 mask 写入）。**

**Gap 4: 后台 worker 未完成最后一段 resident commit**

8 级 staged commit 管线已建好，但 `batch_queue → resident set` 最后一段仍在前台 token-step hook 执行。
这意味着没有真正的 prefetch/compute overlap。

**→ 需要：将 final resident commit 下沉到后台 worker。**

**Gap 5: quantized PIM 路径未接入主 MoE 推理链路**

当前 `PIMMoEBackend` 只走 fp32 的 `expert()` / `linear()` 路径。
`PIMQuantizedRuntime` 存在但只在独立 benchmark 中使用，未集成到 `PIMMoEBackend`。

**→ 需要：让 PIMMoEBackend 在有 GPTQ 权重时自动走 quantized PIM 路径。**

## T-MAC 论文核心思路与 PIM 适配分析

### T-MAC 核心算法

传统 W4A32 matvec：
```
for each weight_nibble q, activation x:
    acc += dequant(q) * x        # 需要乘法
```

T-MAC 位串行分解：
```
# 预计算：对每组 g 个权重的每个 bit 位置，枚举所有 2^g 种组合的部分和
# 推理：用 activation 切片索引 LUT，只做加法和移位
for each bit_position b in [0, bits):
    partial_sum = LUT_b[activation_group_index]
    acc += partial_sum << b
```

### 为什么 T-MAC 特别适合 UPMEM DPU

| DPU 特性 | 传统 MAC | T-MAC LUT |
|----------|---------|-----------|
| 无硬件乘法器 | 每个 `x * w` 需要软件乘法（~10 cycles） | 纯 mram_read + 整数加法（~1 cycle） |
| 有限 WRAM（64KB） | LUT 小：4-bit → 每组仅 16 条 int16 = 32B | 极度友好 |
| 高 MRAM 带宽 | 权重+激活都要读 | 权重已编码在 LUT 索引中，减少读取 |
| 16 tasklet 并行 | 乘法序列化 | 表查找天然并行 |

### 具体适配方案：kernel_mode=6（T-MAC bit-serial LUT for DPU）

**预计算（host 端，一次性）：**
1. 将 4-bit packed qweight 拆成 4 个 1-bit 平面
2. 对每个 bit 平面，按 group_size 分组
3. 每组 g 个权重位，枚举 2^g 种输入组合，预计算部分和 → int16 LUT
4. 将 LUT 与 bit 平面一起 preload 到 DPU MRAM（常驻）

**推理（DPU 端，每次调用）：**
```c
// 对每个 output row pair:
for (bit = 0; bit < 4; bit++) {
    for (group = 0; group < num_groups; group++) {
        // 用 activation 切片作为 LUT 索引
        idx = pack_activation_bits(input_i8[group*g .. (group+1)*g]);
        partial_sum = bit_lut_mram[bit][row][group][idx];  // 纯查表
        acc += partial_sum << bit;
    }
}
```

**关键优势：**
- 内循环零乘法
- LUT 可常驻 MRAM（配合 preload/infer 机制）
- batch 扩展只增加 activation 读取，LUT 复用
- group_size=4 时 LUT 仅 16 条/组，极度 WRAM 友好

## 优先级排序与执行建议

| 优先级 | 改动 | 预期收益 | 复杂度 |
|--------|------|----------|--------|
| **P0** | Quantized preload/infer（qweight+LUT 常驻 MRAM） | 消除量化路径每次调用的权重传输 | 中 — 参照 fp32 preload 模式 |
| **P0** | Quantized PIM 接入主 MoE 链路 | 让真实 GPTQ 权重走 PIM | 中 — PIMMoEBackend 增加分支 |
| **P1** | T-MAC bit-serial DPU kernel (kernel_mode=6) | 解决 batch>1 性能悬崖 + 消除 DPU 软件乘法 | 高 — 新 kernel + host 预计算 |
| **P2** | Batched GPU module injection | 减少 decode Python 开销 10-20% | 低 |
| **P3** | 后台 worker final resident commit | 实现真正 overlap | 中 |
