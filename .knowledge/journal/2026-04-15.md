# 开发日志 2026-04-15

## 完成的工作

- 新增 `pim_native/` 原生桥接代码：DPU linear kernel、host bridge 和构建脚本
- 新增 `pim_linear_runtime.py`，用共享库方式把 Python 主链路接到真实 DPU 线性计算
- 将 `PIMMoEBackend` 从纯 `pim_shadow` 扩展为支持实验性 `pim` 模式
- 在真实硬件上验证了 DPU linear primitive 与 CPU `F.linear` 对齐
- 新增硬件相关测试 `tests/test_pim_runtime.py`

## 当前结论

- 主推理链路里已经有真实 DPU 数值计算参与，不再只是 shadow 统计
- 当前真正上 DPU 的部分是 expert 的线性投影
- SiLU / gating 仍在 host 端执行，因此还不是完整 expert MLP 全上 DPU
- 当前默认只对小 flattened batch 启用真实 PIM，prefill 大批次仍会回退 CPU

## 下一步

- 扩大真实 PIM backend 的 batch 支持范围
- 继续减少 host 参与比例
- 在真实 Qwen3 decode 路径上测量 `cuda_cpu_offload` 与 `cuda_pim` 的差异

## 追加记录

- <!-- updated: 2026-04-15 05:22 --> 为 `PIMMoEBackend` 增加 fused expert kernel 变体，并接入 `benchmark_inference.py` / `example.py`
- <!-- updated: 2026-04-15 05:22 --> 发现 fused expert 第一版的性能问题来自错误的数据流：hidden 被按输出行重复重算
- <!-- updated: 2026-04-15 05:22 --> 已将 fused kernel 重写为“hidden 先落 WRAM，再统一做 down projection”，真实 microbench 改善明显：
  - `linear3 rank_count=4`: `0.0346s`, cycles sum `6982944`
  - `fused rank_count=1`: `0.2921s`, cycles `103175424`
  - `fused rank_count=4`: `0.2965s`, cycles `101454672`
- <!-- updated: 2026-04-15 05:22 --> 结论：fused 已从完全不可用变成可工作的实验路径，但现阶段仍不如 `linear3`，默认策略不应切换
- <!-- updated: 2026-04-15 05:48 --> 继续把 fused expert host bridge 改成二维分片。新的单 expert microbench：
  - `fused rank_count=1, (1,2048,768,2048)`: `0.2668s`, cycles `70500960`, `active_dpus=60`
  - `fused rank_count=4, (1,2048,768,2048)`: `0.2580s`, cycles `70100848`, `active_dpus=60`
  - 对比 `linear3 rank_count=4`: `0.0986s`, cycles `4081328`
- <!-- updated: 2026-04-15 05:48 --> 结论更新：二维分片确实进一步压低了 fused 的单 expert 耗时，但离超过 `linear3` 还有明显差距；当前不应把 `pim_kernel_variant` 默认切到 `fused`
- <!-- updated: 2026-04-15 06:04 --> 目标切换：从“静态 GPU/CPU/PIM 专家放置 + PIM kernel 优化”转到“非专家层常驻 GPU、专家在 GPU/PIM 间动态迁移并与传输 overlap”

  - `GPU -> PIM/CPU`：从 `gpu_experts` 删除模块并更新 mask
- <!-- updated: 2026-04-15 06:58 --> 这一版已经是最小可执行迁移数据面，但仍是同步、逐 expert 的实现，还没有异步预取，也没有和计算 overlap

- <!-- updated: 2026-04-15 07:06 --> 当前仍要明确：这只是“prefetch to CPU staging”，不是最终的 GPU<->PIM 异步迁移，更不是完整 overlap pipeline

- <!-- updated: 2026-04-15 07:14 --> 当某层 GPU resident experts 已满时，系统会先在非活跃 resident experts 里按 hotness 选最冷 victim，执行运行时 eviction，再为新的热点 expert 腾位
