### 2. PIM + MoE Offloading 研究综述

用 web_search + web_fetch 调研了与本项目最相关的 9 篇论文：Fiddler / MoE-Infinity / HOBBIT / HybriMoE / fMoE / PIMoE / NeuPIMs / AttAcc / PIM-AI / P3-LLM。

最有价值的发现：

| 优先级 | 来源 | 创新点 | nano-ktrans 改造点 |
|--------|------|--------|-------------------|
| 🥇 P1 | HybriMoE | MRS score-aware 缓存 | `update_hotness` 加 router scores；warm/activated victim 用 MRS 分数 |
| 🥈 P2 | fMoE | Iteration-level 激活追踪 + prompt 语义预取 | 新增 `ExpertMapStore`（约 200MB）+ background 搜索线程 |
| 🥉 P3 | Fiddler | Cost model：CPU 就地算 vs 拷贝到 GPU | 升级 `pim_prefill_token_threshold` 为 `BackendCostModel` |
| 🏅 P4 | HOBBIT | 混合精度 offloading，cache-miss 用低精度 | 双版本权重（fp16 + int8），风险中高 |
| 🏅 P5 | NeuPIMs | Sub-batch interleaving | 待 batch>1 后再做 |
| 💡 P6 | PIMoE | Throttle-aware 调度 | `PIMLinearRuntime` 暴露 DPU 利用率 |

特别值得注意的一点：**HybriMoE 的 MRS 公式 `S = α · TopP(s) + (1−α)·S`** 直接对应我们当前 `hotness_decay` 参数，但我们丢掉了 router 的概率信息，只留下"是否激活"的二值信号。这恰好是 `cold_promotion_penalty` 指标反映的问题根因。

**fMoE 的 Pearson 实验**证明语义嵌入与专家命中率显著相关，意味着我们现在决定 `decode_require_prefetch_ready` 的开关是否开启，可以不再靠静态 profile，而是根据 prompt 语义与历史 store 的相似度动态决定。

## 4. P1 + P2 落地实现（同日追加）

用户要求"实现 P1 和 P2"，直接进入编码。最终 21 文件改动 +764/−79，零 lint 错误。

### 4.1 P1 — MRS Score-Aware Hotness

**公式**：`S = α · TopP(router_scores) + (1 − α) · S`（HybriMoE §IV.C）

**实现**：
- `utils/expert_runtime_state.py::update_hotness` 新增 3 个 kwargs：
  `router_scores` / `mrs_alpha` / `top_p`；
  所有都为 `None` 时保持旧 bincount 行为，**完全向后兼容**。
- 关键实现细节：
  - MRS 模式下先按 `top_p` 对 router scores 做裁剪（HybriMoE 默认 `p = 2·top_k`）
  - 用 `torch.scatter_add_` 而不是 `bincount`（后者不支持浮点 weight）
  - **除以 token 数归一化**：否则长 prompt 会在单次 observe 里把 EMA 推到接近 1
  - `mrs_alpha` 范围校验：必须在 `(0, 1]`
- `SchedulerConfig` 新增 `hotness_mrs_alpha` / `hotness_top_p` 字段
- `DynamicExpertScheduler.observe` 新增可选 `topk_weights`；
  MRS 配置开启但 `topk_weights=None` 时自动 fallback 到 bincount 并记入 `hotness_bincount_observations`
- `HybridMoE.forward` 中将已经算好的 `topk_weights`（经 router softmax + top_k）直接传给 scheduler

**启用**：`LLM(scheduler_hotness_mrs_alpha=0.5, scheduler_hotness_top_p=4, ...)`

### 4.2 P2 — Expert Map Store + Prompt 语义预取

**新建** `nano_ktrans/utils/expert_map_store.py`（约 250 行）：

- `ExpertMap` dataclass：单次 iteration 的指纹，含 L2-normalized prompt embedding
  + `{layer_idx: (raw_probs, unit_probs)}`
- `ExpertMapStore`：
  - LRU 容量管理（`OrderedDict` + `popitem(last=False)`）
  - 两阶段搜索：
    - **Semantic phase**（`layer_idx < prefetch_distance`）：prompt embedding 余弦
    - **Trajectory phase**（之后）：已观测 gate 分布的平均余弦
  - 线程安全 `RLock`（为将来挂到 background worker 预留）
  - 诊断：commit_count / eviction_count / {semantic,trajectory}_{queries,hits}

**集成**：
- `HybridMoE` 新增 `expert_map_store` + `expert_map_prefetch_top_k` 构造参数
- forward 中调用 `_record_router_probs`（每层记 gate 均值分布）
  + `_request_map_store_prefetch`（从 store 拿预取建议走同一 `_request_prefetch` 漏斗）
- `MixtralModel.forward` 负责 iteration 级 begin/commit；
  用 **token embedding 的均值** 作 prompt 语义锚点（fMoE §5.1），无需额外模型
- 全链路：`LLM` → `MixtralForCausalLM` → `MixtralModel` → `MixtralDecoderLayer` → `HybridMoE`

**启用**：`LLM(enable_expert_map_store=True, expert_map_store_capacity=1024, ...)`

### 4.3 踩坑与经验

**坑 1：`update_hotness` 不做 token 数归一化会炸 EMA**
prefill 阶段一次 observe 可能看到 512 个 token，若直接累加 scatter_add，hotness 会被单次请求推到量级极大，后续 decode 的 `alpha · new + (1-α) · old` 中 old 占绝对主导，EMA 失效。
→ 最终实现按 token 数归一化，保证单次 observe 的 `score_mass ∈ [0, 1]`。

**坑 2：`@lru_cache` 对 dict 参数会 TypeError**
修 P4（RoPE scaling）时才发现原先 `get_rope(rope_scaling=...)` 的 `@lru_cache` 缓存在传 dict 时会崩。
→ 拆成 `_validate_rope_scaling`（不缓存）+ `_build_rope(@lru_cache)`（只缓存 hashable 部分）。**经验：给 lru_cache 函数加 dict 参数之前必须确认该参数永不进入 cache key**。

**坑 3：scheduler.observe 的向后兼容**
现有所有测试都不传 `topk_weights`，直接加必填参数会破坏一堆测试。
→ `topk_weights: torch.Tensor | None = None` + 运行时判断；MRS 未开启或 weights 为 None 时走旧 bincount 分支，并单独计数到 `hotness_bincount_observations`。经验：**所有新增诊断必须同时记录"已切到新路径"和"仍在旧路径"两个计数**，否则 benchmark 没法判断新代码是否真的生效。

**坑 4：Expert Map Store 的 prompt 锚点**
第一版想用 `hidden_states[0]`（第一个 token），但 BOS token 的 embedding 在不同 prompt 里几乎相同，完全没有区分度。
→ 改成 `embed_tokens(input_ids).mean(dim=(0, 1))`，跨所有 token 取平均，足以区分不同 prompt 的主题，且无需额外 forward。

**坑 5：HybridMoE 的 `record_router_probs` 放置位置**
一开始放在 `with self._pipeline_lock:` **之内**，但这会让 record_router_probs 阻塞在 pipeline_lock 上，而 lock 的主要目的是保护 migration 状态。
→ 挪到 lock 外：record 只改 in-flight ExpertMap（per-iteration 对象，单线程），不触及任何共享 pipeline 状态。

**经验总结**（写进 glossary / gotchas）：
1. MoE 系统里"激活频率"与"路由 confidence"是两个信号，HybriMoE 的启示是后者能独立影响 cache 命中率
2. Expert offloading 框架的"预取漏斗"最好只有一个（`_request_prefetch`），多来源（dynamic scheduler / map store / prefill policy）只向该漏斗投递候选，复用同一套 cache/budget/lifecycle
3. 任何"按 iteration 积累历史"的结构（EAM、Expert Map）都要有 LRU 容量上限，否则长对话会无界增长
4. prompt 语义特征对 MoE 路由预测有效，但要做 L2 normalize 才能用纯内积做余弦
5. `torch.scatter_add_` 比 `bincount` 更通用：支持浮点 weight，适合把 router probability mass 写进 EMA
