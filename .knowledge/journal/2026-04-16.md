# 开发日志 2026-04-16

## 完成的工作

- 完成 PIM 参数常驻优化方案的完整架构设计
- 详细分析了当前 PIM 后端的性能瓶颈：每次推理都将专家权重从 CPU RAM 重新传输到 DPU MRAM
- 确认 DPU expert kernel 的 `__mram_noinit` MRAM 数组天然支持跨 launch 持久化，无需修改 DPU kernel

## 当前结论

- 核心瓶颈在 `host_expert_bridge.c::pim_expert_run()`：每次调用都无条件传输 gate/up/down 三个权重矩阵
- 对于 Qwen3 级别模型，每次调用传输 ~670MB 权重，而实际 input 仅 ~16KB，比例约 44000:1
- CPU 后端（`AMXInt4_MOE`）已有正确的做法：权重加载一次后常驻在 C++ 对象中，推理仅传输 activation
- 解决方案：将 C bridge 拆分为 `preload`（权重传输）和 `infer`（仅 activation 传输）两个阶段

## 方案设计要点

### C Bridge 层 (`host_expert_bridge.c`)
- 新增 `pim_expert_preload()`: 权重传输 + sharding，带 expert_id 缓存命中检测
- 新增 `pim_expert_infer()`: 仅传输 input activation，使用已缓存的 sharding geometry
- 新增 `pim_expert_evict()`: 清除常驻状态（迁移时使用）
- 现有 `pim_expert_run()` 改为 `preload() + infer()` 的包装，保持向后兼容

### Python Runtime 层 (`pim_expert_runtime.py`)
- 新增 `preload()` / `infer()` / `evict()` 方法及 ctypes 绑定
- Python 侧缓存已 pad 的 f32 权重张量，避免重复转换
- Python 侧镜像常驻 expert_id，减少 ctypes 调用开销

### PIMMoEBackend 编排层 (`pim_moe.py`)
- `_run_expert_fused_on_dpu()` 改为先 `preload` 再 `infer`
- 专家排序优化：当前已常驻的专家排到最后处理，最大化下次 token 命中率
- Prefill 结束时预测性预加载最热 CPU 专家

### 与专家迁移系统集成
- PIM→GPU 迁移时 evict DPU 常驻状态
- CPU→PIM 迁移时在 lifecycle `WARMED` 阶段触发 preload

## 预期性能提升

- 缓存命中时：权重传输从 ~670MB 降为 0，PIM 总线时间从 ~50-100ms 降至 ~0.1ms
- 缓存未命中时：退化为当前行为，无额外开销
- DPU 计算时间不变，仅消除数据传输开销
