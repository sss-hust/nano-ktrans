# 开发日志 2026-04-19

## 完成的工作

- 将真实 `Qwen/Qwen3-30B-A3B-GPTQ-Int4` 权重接入 operator-only benchmark，确认 `gate/up/down` 的真实 GPTQ tensor layout 可直接驱动 PIM quantized runtime。
- 为 quantized PIM 路径补齐 `transfer_only / unpack_only / dequant_only / full` 分项 profiling，并进一步拆出权重常驻加载与稳态执行时间。
- 引入 `kernel_mode=4` 的最小整数化原型：host 端将激活量化为 `int8`，按组构造 `int16` dequant LUT，DPU 侧执行 `int8 x int16 -> int32` accumulate。
- 将 `kernel_mode=4` 收敛为稳定主线，移除不稳定的 `int16` 原型，并补齐真实 DPU 的 `batch=4` 回归。
- 对 `kernel_mode=4` 做真实 `FIXED_BATCH_TILE=1/2/4/8` sweep，确认 `down batch=4` 受 shape-gated fallback 保护、接近 CPU grouped，而 `gate batch=4/8` 尚未找到稳定翻盘的 tile 配置。

## 当前结论

- 原始 `W4A32 + soft-float dequant` 路径的主瓶颈在 DPU 侧 unpack/dequant/compute，本质上不是 host 传输问题。
- `kernel_mode=4` 证明“更整数化的数据流”是对的：在 `batch=1` 的真实 GPTQ `gate/down` 上，PIM 已能超过 CPU grouped baseline。
- 但这种优势强依赖小 batch；`gate batch>1` 仍明显吃亏，当前的 batch-tile 只改变常数项，还不足以把大输入维度 case 稳定拉回 CPU 之上。

## 下一步

- 继续围绕 `kernel_mode=4` 优化 `gate batch>1` 的数据流，重点不再是盲目增大 tile，而是减少大输入维度下的 batch 组织与提交开销。
- 若后续批量路径仍无稳定收益，则应把 `kernel_mode=4` 明确定位为 decode 小 batch 优先的 PIM 算子路径，而不是通用 matvec 加速器。
